# 目的

帮助用户持续优化全局上下文（`~/.claude/CLAUDE.md`），使其保持精简、高效、符合官方标准。

这个 skill 做三件事：
1. 从用户输入中提取适合全局生效的内容，生成具体的改动
2. 对不适合全局的内容，建议是否值得抽为独立 skill
3. 按官方标准审查和精简现有全局上下文

官方标准来源（需要时重新获取最新版本）：
- https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices
- https://code.claude.com/docs/en/best-practices

# 与 nix-self-iteration-protocol 的区别

- **nix-self-iteration-protocol**：诊断已发生的 prompt failure，分类失败模式，修复指令系统
- **本 skill**：从用户输入或官方标准出发，主动优化全局上下文的内容和结构

如果用户描述的是"某条规则没生效"或"skill 误触发"，用 nix-self-iteration-protocol。
如果用户想的是"我的全局上下文应该怎么写更好"，用本 skill。

# 输入模式

用户可能以四种方式触发：

## 模式 1：描述协作摩擦

用户说最近遇到的不顺心场景。

做法：
1. 判断摩擦的根因——是全局上下文缺规则，还是 skill 缺失、tooling 缺口、或模型能力边界
2. 如果是全局上下文问题，提出具体的规则改动
3. 如果不是，告诉用户真正的修复点在哪里

## 模式 2：评估改动提案

用户直接给出想加入全局上下文的内容。

做法：
1. 读取当前 `~/.claude/CLAUDE.md`
2. 用评估清单逐条检查提案内容（见下方「评估标准」）
3. 对每条内容给出判定：适合全局 / 应抽为 skill / 不需要
4. 对适合全局的内容，给出改写后的具体文本和插入位置
5. 对建议抽为 skill 的内容，说明理由和 skill 的大致轮廓

## 模式 3：审查现有全局上下文

用户没提供特定内容，要求直接优化。

做法：
1. 读取当前 `~/.claude/CLAUDE.md`
2. 用评估清单逐条审查每条现有规则
3. 输出审查报告：保留 / 精简 / 删除 / 迁移到 skill，每条附理由
4. 给出改写后的完整全局上下文

## 模式 4：从信息中提取

用户给一个指令让你获取信息（如读某个文件、搜索某个话题），然后从中提取全局上下文候选内容。

做法：
1. 执行用户的指令，获取信息
2. 从信息中筛选出可能对全局上下文有价值的内容
3. 按模式 2 的评估流程处理

# 评估标准

对每条候选规则，依次检查：

1. **跨任务性** — 是否所有/绝大多数任务类型都需要？如果只在特定工作流中需要，它属于 skill
2. **不可推断性** — Claude 能否从代码或上下文自行推断？能推断的不需要写
3. **可观察性** — 违反时能否在输出中看到？不可观察的规则无法验证也无法改进
4. **非冗余性** — 是否与 Claude 的默认行为不同？重复默认行为的规则是噪音
5. **精简性** — 是否已是最短的有效表述？
6. **带动机性** — 是否解释了 why？解释 why 比 MUST/NEVER 更有效（Claude 4.5/4.6 对系统提示很敏感，不需要重锤指令）
7. **无矛盾性** — 是否与其他规则冲突？

# 官方标准参考

## 全局 CLAUDE.md 精简原则

来源：https://code.claude.com/docs/en/best-practices

- 把 CLAUDE.md 当代码对待：review、prune、test
- 如果删掉某行不会导致 Claude 犯错，就删掉它
- 过长的 CLAUDE.md 会导致规则被忽略（官方明确的 failure pattern: "Over-specified CLAUDE.md"）

## 提示词工程标准

来源：https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices

### 全局上下文适合的内容

1. **角色设定** — 聚焦行为和语气的单句角色描述
2. **清晰直接的指令** — 无歧义、可观察行为的规则
3. **带动机的约束** — 解释 why 比 MUST/NEVER 更有效
4. **输出格式偏好** — 只在偏好与默认值不同时才写
5. **过度工程化约束** — Claude 4.5/4.6 有过度工程化倾向，全局层适合加此类约束

### 全局上下文不适合的内容

1. **专项工作流** — 应放入 skill（按需加载，不占全局 token）
2. **具体到单个项目的规则** — 应放项目级 CLAUDE.md
3. **可由 tooling 确定性约束的** — 用 hooks、linters、tests 替代
4. **过度指令** — Claude 4.5/4.6 对系统提示更敏感，不需要 CRITICAL/MUST 等强调词
5. **冗余规则** — 与 Claude 默认行为一致的规则是噪音
6. **可从代码推断的** — 架构、文件路径、代码风格等 Claude 能自己看出来的

### Claude 4.5/4.6 行为特性（影响全局上下文设计）

- 更直接、更少冗余输出
- 对系统提示更敏感，回调过度强调的指令
- 有过度工程化倾向
- 默认使用 adaptive thinking
- 擅长并行工具调用
- 自然地使用 subagent 委托

# 输出格式

无论哪种输入模式，最终输出都包含：

## 1. 全局上下文改动

对每条改动：
- 改动类型：新增 / 修改 / 删除
- 改动内容：具体的改写后文本
- 评估依据：引用评估清单中的哪些条目
- 在现有结构中的位置：应插入/替换 CLAUDE.md 的哪个 section

## 2. Skill 建议（如有）

对每个建议的 skill：
- 名称和一句话描述
- 它覆盖的场景
- 为什么不适合放在全局上下文（通常是因为不满足跨任务性）

## 3. 不采纳的内容（如有）

对每条不采纳的内容，说明原因（冗余、可推断、属于 tooling 层等）。

# 执行纪律

- 读完现有全局上下文再提建议，不要凭空写
- 全局上下文是寸土寸金的——每加一条规则，都在消耗每次对话的 token 预算。宁可少一条，不可多一条
- 不要重复 Claude 的默认行为（比如"要写安全的代码"——Claude 默认就会这么做）
- 解释 why 而不是用大写字母喊 MUST。Claude 4.5/4.6 理解动机比服从命令更可靠
- 改动要具体到可以直接 apply 的文本，不要只给抽象建议
